# Rare Pick 프로젝트 리서치 보고서

## 1. 조사 범위와 결론

이 저장소는 겉보기에는 단순한 Next.js 앱처럼 보이지만, 실제로는 다음 네 축이 공존하는 혼합형 구조다.

- `app/`, `components/`, `lib/` 중심의 현재 메인 Next.js 앱
- `collector/` 중심의 가격 수집 배치와 감시 워커
- `render-api/`의 별도 Node HTTP API
- `cloudflare/click-worker/`의 별도 Cloudflare Worker API

핵심 결론은 다음과 같다.

1. 현재 사용자 진입점은 `app/page.js` 하나이며, 메인 화면은 상품 목록 대시보드가 아니라 URL 기반 감시 등록 화면이다.
2. 실제 감시 기능의 본체는 `watch_jobs` 테이블과 `collector/run-watch-worker.js`이며, Next 앱만 띄워서는 감시/알림이 완결되지 않는다.
3. 문서와 네이밍은 Amazon + Coupang 지원처럼 보이지만, 현재 실제 사용자 플로우는 Amazon 중심이며 Coupang은 사실상 중단 상태다.
4. 저장소 안에 활성 구현, 대체 구현, 레거시 구현이 함께 남아 있어 구조를 잘못 읽으면 실제 동작을 오해하기 쉽다.

## 2. 루트 구조 요약

주요 디렉터리와 역할은 아래와 같다.

- `app/`
  - Next App Router 엔트리
  - 실제 활성 페이지와 API route handler 보유
- `components/`
  - 클라이언트 UI 컴포넌트
  - 현재는 `watch-dashboard.js`가 실사용, `product-dashboard.js`는 미연결
- `lib/`
  - DB 래퍼와 앱 서비스 계층
  - watch 생성, 상품 통계, URL 파싱 담당
- `collector/`
  - 가격 수집기와 watch worker
  - HTML 스크레이핑, 환율 처리, 알림 발송, DB 기록 포함
- `db/`
  - PostgreSQL 스키마
- `render-api/`
  - 별도 Node 서버형 API
- `cloudflare/click-worker/`
  - Supabase REST 기반 Worker API
- `legacy-api/`
  - 현재 App Router 바깥에 남아 있는 예전 Next route 구현
- `docs/`
  - 실제 상태 문서라기보다 “권장 구조” 문서

## 4. 현재 메인 앱의 실제 동작

### 4.1 진입점

현재 메인 페이지는 `app/page.js`이며 다음 한 줄이 사실상 전부다.

- `WatchDashboard`를 렌더링한다.
- 서버에서 DB 데이터를 미리 읽어오지 않는다.
- `initialWatches={[]}`가 고정이라 첫 렌더 기준 목록은 항상 빈 상태다.

즉, 현재 화면 구조는 “서버가 껍데기를 렌더링하고, 실제 상태는 브라우저가 관리하는 형태”다.

### 4.2 레이아웃

`app/layout.js`는 최소한의 HTML 셸만 제공한다.

- `<html lang="ko">`
- 전역 CSS import
- title/description 메타데이터

흥미로운 점은 메타데이터 설명이 아직 “쿠팡/아마존 가격 추적 및 제휴 링크 대시보드”인데, 실제 메인 UI는 상품 대시보드가 아니라 watch 등록 화면이라는 점이다.

### 4.3 메인 UI: `components/watch-dashboard.js`

이 파일이 현재 제품의 핵심 UI다. 담당 역할은 아래와 같다.

- 입력 폼 상태 관리
- URL 사전 검증
- `/api/watch` 또는 외부 API 호출
- 실패 시 브라우저 `localStorage` fallback
- 등록 목록 렌더링

핵심 로직:

1. 초기 마운트 시 `localStorage`의 `rare-pick-watches-v1`를 읽음
2. `watches` 상태가 바뀔 때마다 다시 `localStorage`에 저장
3. 사용자가 URL/목표가/이메일을 입력 후 제출
4. `parseProductUrl()`로 URL 파싱
5. `POST /api/watch` 호출 시도
6. 서버 성공이면 응답 watch를 목록 앞에 추가
7. 서버가 5xx이거나 네트워크 예외면 로컬 watch 객체를 만들어 목록에 추가

중요한 해석:

- README는 “DB 실패 시 fallback”처럼 설명하지만, 실제 UI는 항상 `localStorage`를 읽고 항상 다시 저장한다.
- 따라서 `localStorage`는 백업이 아니라 실질적인 상태 저장소 역할도 수행한다.
- 서버에 저장 성공한 항목도 다시 `localStorage`로 복제된다.

### 4.4 실제 사용자 경험

현 상태에서 사용자가 보는 경험은 아래와 비슷하다.

1. 첫 방문 시 목록은 SSR 기준 빈 상태
2. 클라이언트 hydration 후 `localStorage`에 저장된 감시 목록이 있으면 표시
3. 새 감시 등록 시 서버 저장을 우선 시도
4. 실패해도 화면상으로는 로컬 목록에 추가되어 “등록된 것처럼” 보임

이 구조 때문에 현재 화면의 진실 원천(source of truth)은 DB가 아니라 브라우저일 가능성이 높다.

## 5. Next API 레이어

현재 활성 Next route handler는 세 개뿐이다.

- `app/api/watch/route.js`
- `app/api/click/route.js`
- `app/api/health/route.js`

### 5.1 `POST /api/watch`

역할:

- `productUrl` 검증
- `notifyEmail` 형식 검증
- `DATABASE_URL` 존재 여부 확인
- `createWatchJob()` 호출

특징:

- DB가 없으면 `503`
- 클라이언트는 `>= 500`을 fallback 조건으로 보기 때문에, `503`도 로컬 저장 fallback을 유발한다
- 내부에서 단순 insert만 하는 것이 아니라 Amazon이면 즉시 초기 가격 수집까지 시도한다

즉, 이 엔드포인트는 “가벼운 등록 API”가 아니라 “DB insert + 외부 스크레이핑”을 포함한 느린 API다.

### 5.2 `POST /api/click`

역할:

- 숫자형 `productId` 검증
- `click_events` insert

특징:

- DB 미설정이면 `insertClickEvent()`가 조용히 반환
- watch 등록과 달리 브라우저 fallback 저장은 없다
- 현재 이 엔드포인트를 쓰는 `ProductDashboard`가 메인 페이지에 연결되어 있지 않아 사실상 휴면 상태다

### 5.3 `GET /api/health`

역할:

- DB 미설정이면 `{ ok: true, database: 'disabled' }`
- DB 설정 시 `SELECT 1`
- 실패 시 500 반환

현재 프론트 UI가 직접 사용하지는 않지만 Render 헬스체크에 적합한 형태다.

## 6. 앱 서비스 계층

### 6.1 `lib/db.js`

간단한 PostgreSQL 래퍼다.

- `hasDatabase()`: `DATABASE_URL` 존재 여부만 확인
- `getPool()`: 없으면 throw, 있으면 singleton Pool 생성
- `query()`: Pool query 호출

주의점:

- collector 쪽 DB 접근과 정책이 다르다
- 앱 쪽은 DB가 없으면 하드 실패에 가깝다
- collector 쪽은 DB가 없어도 일부 기능을 소프트 스킵한다

### 6.2 `lib/url-parser.js`

입력 URL에서 source/externalId를 뽑는 로직이다.

- Amazon: `dp`, `gp/product`, `product` 패턴에서 ASIN 추출
- Coupang: `/vp/products/<id>` 패턴 추출

하지만 실제 반환 정책은 다음과 같다.

- Amazon은 허용
- Coupang은 ID를 파싱한 후에도 “임시 보류” 에러를 던짐
- 그 외 URL은 “현재는 amazon URL만 지원” 에러

즉, 코드 상으로는 Coupang 파서를 갖고 있지만 실제 사용자 입력은 Amazon만 통과한다.

### 6.3 `lib/watch-service.js`

watch 등록 핵심 서비스다.

주요 함수:

- `createWatchJob()`
- `tryCollectInitialAmazonPrice()`
- `listWatchJobs()`

`createWatchJob()` 동작:

1. DB 존재 확인
2. `parseProductUrl()`로 URL 파싱
3. `products` upsert
4. `watch_jobs` insert
5. source가 Amazon이면 `tryCollectInitialAmazonPrice()` 실행
6. 방금 생성된 watch를 다시 조회하여 DTO 반환

`tryCollectInitialAmazonPrice()` 동작:

1. `fetchAmazonPrice()` 호출
2. 상품 메타데이터(title, image, category) 업데이트
3. `price_history` insert
4. `watch_jobs.last_checked_at`, `last_price`, `last_error` 갱신
5. 실패해도 watch 생성은 유지하고 `last_error`만 남김

`listWatchJobs()`는 매우 중요한데, 현재 메인 페이지에서는 사용되지 않는다. 이 함수는 원래 아래 데이터를 제공할 준비가 되어 있다.

- watch 기본 정보
- 상품 title
- price_history 기반 `lowestPrice`

즉, 서버 truth 기반 목록 API 조각은 이미 존재하지만 현재 메인 진입점에 연결되지 않았다.

### 6.4 `lib/products-service.js`

이 파일은 예전 상품 대시보드 흐름을 담당한다.

함수:

- `listProductsWithStats()`
- `getPriceHistory()`
- `insertClickEvent()`

특징:

- DB가 없으면 `mock-data.js` 사용
- 상품/가격 히스토리 대시보드에 필요한 형태의 데이터 가공
- 현재 메인 페이지에서는 사용되지 않음

즉, 이 서비스는 “현행 제품 흐름”보다 “과거 대시보드형 MVP”의 흔적에 가깝다.

## 7. UI 중 휴면 상태인 제품 대시보드

`components/product-dashboard.js`는 다음을 수행한다.

- 상품 카드 목록 표시
- 7일 가격 추이 그래프 렌더링
- 현재가/최저가/최고가 계산
- 제휴 링크 클릭 시 `POST /api/click`

하지만 현재 `app/page.js`가 이 컴포넌트를 렌더링하지 않으므로 실제 사용자 경로에는 없다.

잠재적 문제점도 있다.

- `products`가 비었을 때 `selectedProduct.id` 접근 위험
- mock 데이터 id는 문자열인데 `POST /api/click`은 숫자를 기대
- `trackClick()`은 응답 실패를 사용자에게 드러내지 않음

## 8. DB 스키마

`db/schema.sql` 기준 핵심 테이블은 여섯 개다.

### 8.1 `products`

상품 마스터 테이블.

- `source`
- `external_id`
- `title`
- `image_url`
- `affiliate_url`
- `category`

`(source, external_id)` 유니크 제약이 있다.

### 8.2 `price_history`

가격 스냅샷 이력.

- `product_id`
- `price`
- `currency`
- `captured_at`

### 8.3 `click_events`

제휴 링크 클릭 로그.

- `product_id`
- `referrer`
- `session_id`
- `clicked_at`

### 8.4 `affiliate_reports`

제휴 성과 집계용 테이블.

현재 코드상 사용 흔적은 거의 없다.

### 8.5 `watch_jobs`

감시 등록의 본체.

- `product_id`
- `source`
- `external_id`
- `product_url`
- `target_price`
- `notify_email`
- `is_active`
- `last_checked_at`
- `last_price`
- `last_error`

### 8.6 `notifications`

알림 발송 결과 기록.

- `watch_job_id`
- `product_id`
- `notification_type`
- `message`
- `payload`
- `status`

전체적으로 보면 `watch_jobs`와 `notifications`가 최근 추가된 기능축이고, `products`/`price_history`/`click_events`/`affiliate_reports`는 기존 가격 대시보드 축이다.

## 9. 가격 수집 배치: `collector/run-price-collector.js`

이 스크립트는 “고정된 추적 대상 목록”을 수집하는 배치다.

동작:

1. `getConfig()`로 추적 목록 읽기
2. 각 item을 순차 수집
3. 결과를 PostgreSQL에 insert
4. NDJSON 파일에 append
5. 콘솔에 로그 출력

핵심 문제:

- `collector/config.js`의 `trackItems`가 DB 기반이 아니라 하드코딩 예시값이다
- Amazon/Coupang 예시 상품 ID가 placeholder라 운영용 입력으로 보기 어렵다
- item 하나 실패 시 전체 배치가 종료된다
- DB insert 이후 NDJSON 저장이라 DB 실패 시 파일 저장도 못 한다

즉, 이 스크립트는 운영 핵심보다는 시드/예제 배치 성격이 강하다.

## 10. 감시 워커: `collector/run-watch-worker.js`

이 프로젝트의 실제 감시 엔진은 이 파일이다.

전체 흐름:

1. `getUsdKrwRate()`로 USD/KRW 환율 확보
2. `listActiveWatchJobs()`로 활성 watch 조회
3. 각 watch를 순차 처리
4. 현재 가격 수집
5. KRW 환산
6. 스냅샷 저장 + watch 상태 갱신
7. 목표가 도달 또는 신저가 여부 판단
8. 알림 발송
9. `notifications` 기록

핵심 판단 로직:

- 목표가 비교는 KRW 기준
- 신저가 판단은 원본 통화 기준 최소값과 비교

이 구조는 나름 일관되지만, 데이터 해석 시 주의가 필요하다.

- `watch_jobs.last_price`와 `price_history.price`는 원본 통화 그대로 저장
- 목표가는 KRW 기준으로 비교

즉, Amazon USD 상품의 경우 저장값은 USD인데 사용자 목표가는 KRW일 수 있다.

### 10.1 소스 지원 상태

`fetchCurrentPrice()` 기준:

- Amazon: 지원
- Coupang: `temporarily paused` 에러

따라서 현재 watch worker는 실질적으로 Amazon 전용이다.

## 11. 수집/저장 레이어: `collector/repository.js`

collector 전용 DB 레이어다.

주요 함수:

- `appendSnapshots()`
- `insertSnapshotsToDb()`
- `listActiveWatchJobs()`
- `saveSnapshotForWatch()`
- `markWatchError()`
- `createNotificationEvent()`

### 11.1 앱 DB 레이어와의 차이

`lib/db.js`와 달리 `collector/repository.js`는 `DATABASE_URL`이 없으면 `null`을 반환한다. 즉:

- 앱/API는 DB 없음 = 실패
- collector는 DB 없음 = 일부 기능 skip

운영 관점에서 이 차이는 중요하다.

### 11.2 `saveSnapshotForWatch()`

watch 처리 시 핵심 트랜잭션.

1. `products` upsert
2. 기존 최저가 조회
3. `price_history` insert
4. `watch_jobs` 상태 업데이트

반환값:

- `productId`
- `previousMinPrice`
- `latestPrice`

이 반환값을 worker가 알림 판단에 사용한다.

## 12. HTML 수집기 공통부: `collector/fetchers/html-price.js`

이 파일은 이름보다 더 중요한 공통 엔진이다.

제공 기능:

- HTML fetch
- 브라우저/HTTP 전략 전환
- JSON-LD 가격 추출
- regex 기반 가격 추출 보조

전송 모드:

- `browser-first` 기본
- `browser-only`
- `http-only`

`browser-first`인 경우:

1. Playwright 브라우저로 시도
2. 실패 시 일반 HTTP 요청으로 폴백
3. 둘 다 실패하면 합성 에러 반환

즉, 수집 안정성 확보를 위해 브라우저와 HTTP를 모두 활용하도록 설계돼 있다.

## 13. Amazon fetcher

파일명은 `amazon-paapi.js`지만 실제로 PAAPI를 쓰지 않는다. 본질은 HTML 스크레이퍼다.

수행 내용:

- 가격 추출
- 통화 판별
- 상품명 추출
- 이미지 URL 추출
- 카테고리 추출
- 차단 페이지/비상품 페이지 판별

추출 전략:

- JSON-LD offer
- `a-offscreen`, `priceblock_*`, `priceToPay` 등 regex
- 메타 태그, 제목, breadcrumb 기반 메타데이터 보강

특징:

- 브라우저 결과가 막혔거나 상품 페이지가 아니면 `http-only` 재시도
- 가격은 찾았지만 메타데이터가 빈약하면 다시 `http-only`로 보강 시도

즉, 이 파일은 단순 가격 추출기를 넘어 “안티봇 환경에서 최대한 salvage 하는 스크레이퍼”에 가깝다.

## 14. Coupang fetcher

`collector/fetchers/coupang-openapi.js`도 이름과 다르게 실제는 HTML 스크레이퍼다.

수행 내용:

- 다양한 regex로 가격 추출
- 차단/비상품 페이지 감지
- 필요 시 `http-only` 재시도

하지만 중요한 점:

- 배치 수집기에서는 Coupang fetcher를 호출할 수 있다
- URL 파서는 Coupang 입력을 차단한다
- watch worker는 Coupang을 명시적으로 중단한다

즉, 저장소 안에서 Coupang 지원 상태가 일관되지 않다.

## 15. 환율 처리: `collector/fx-rate.js`

USD -> KRW 환율을 하루 단위 캐시로 관리한다.

우선순위:

1. fresh cache
2. 외부 API
3. stale cache
4. `USD_KRW_RATE` 환경변수 폴백

장점:

- 외부 API 장애에 견딤
- 캐시 파일로 반복 호출 절감

주의:

- 비교는 KRW 기준인데 DB 저장은 원본 통화 그대로라, 데이터 조회 시 통화 컨텍스트를 잃기 쉽다

## 16. 알림 처리: `collector/notifier.js`

알림 채널:

- 웹훅
- 이메일

정책:

- 웹훅 URL이 없으면 콘솔 채널로 간주
- 이메일은 `reason === 'target_price'`일 때만 시도
- Gmail SMTP 정보가 있어야 이메일 발송 가능
- 일부 채널 실패해도 전체 워커는 멈추지 않음

알림 실패는 보통:

- `watch_jobs.last_error`가 아니라
- `notifications.status = 'failed'`

로 남는다.

이 설계는 “수집 실패”와 “알림 실패”를 분리한다는 점에서 괜찮다.

## 17. 배포/실행 모델

### 17.1 현재 주 배포: `render.yaml`

실제로 배포되는 것은 Next 앱 한 개다.

- `npm ci && npm run build`
- `npm run start`
- `NODE_VERSION=22`
- `DATABASE_URL` 외부 주입

즉, README의 “단일 Web Service”는 웹앱 자체 기준으로는 맞다.

하지만 감시/알림 전체 서비스 기준으로 보면 완전하지 않다.

이유:

- `worker:watch`는 Render web service 안에서 자동으로 돌지 않는다
- 별도 크론/스케줄러가 필요하다

따라서 실제 운영 구성은 다음에 더 가깝다.

- Next 웹 서비스
- PostgreSQL
- 정기 실행되는 watch worker
- 선택적 웹훅/이메일

### 17.2 별도 Node API: `render-api/server.js`

이 구현은 순수 `node:http` 서버다.

엔드포인트:

- `GET /health`
- `POST /click`
- `POST /watch`

특징:

- CORS 지원
- `source`, `externalId`, `productUrl`를 클라이언트가 직접 보내는 계약
- `watch_jobs`에 바로 insert
- Next의 `createWatchJob()`처럼 URL 재파싱이나 초기 Amazon 수집을 하지 않음

즉, “기능은 비슷하지만 계약과 동작이 다른 대체 API”다.

### 17.3 Cloudflare Worker: `cloudflare/click-worker/src/index.js`

이 구현은 Supabase REST 직접 호출형 서버리스 API다.

엔드포인트:

- `GET /health`
- `POST /click`
- `POST /watch`

특징:

- DB 직접 연결이 아니라 Supabase REST 사용
- CORS 설정 변수명이 `ALLOWED_ORIGIN`
- `render-api`와 유사하게 얇은 insert API

즉, 이것도 Next API와 동작 계약이 다르다.

## 18. 레거시 구현

`legacy-api/app-api/`는 현재 App Router 바깥에 있어 활성 경로라고 보기 어렵다. 다만 이 안에는 흥미로운 조각이 남아 있다.

- `GET /watch`: `listWatchJobs()` 사용
- `GET /products`: 상품 목록 반환
- `GET /prices/[productId]`: 가격 히스토리 반환
- `POST /click`
- `GET /health/db`

특히 `legacy-api/app-api/watch/route.js`는 현재 메인 UI가 필요로 하는 “watch 목록 읽기” 경로를 이미 가지고 있었다.

즉, 현행 메인 UI가 서버 목록을 못 읽는 이유는 기술 부재가 아니라 “활성 경로 재연결이 안 된 상태”에 가깝다.

## 19. 문서와 실제 구현의 차이

### 19.1 README vs 실제 상태

README는 대체로 방향은 맞지만, 실제 구현은 더 복잡하다.

- 문서상 “쿠팡/아마존”이 강조되지만 실제 입력 경로는 Amazon만 허용
- 문서상 fallback은 보조처럼 보이지만 실제 UI는 `localStorage` 의존이 더 강함
- 문서상 단일 서비스처럼 보이지만 실제 감시/알림 완성에는 별도 워커 실행이 필요

### 19.2 `docs/nextjs-mvp-structure.md`

이 문서는 현재 상태 설명이 아니라 “전환 권장 구조” 문서다.

예시 API:

- `GET /api/products`
- `GET /api/prices/:productId`
- `POST /api/click`

하지만 현재 활성 `app/api`에는 `products`, `prices` 경로가 없다.

즉, 이 문서를 현재 구조 문서로 읽으면 안 된다.

### 19.3 파일명과 실제 동작의 차이

- `amazon-paapi.js`: 실제로 PAAPI 미사용
- `coupang-openapi.js`: 실제로 OpenAPI 미사용

이 네이밍은 미래 계획이나 과거 설계의 흔적으로 보이며, 현재 구현은 HTML 스크레이핑 중심이다.

## 21. 구조적 문제와 리스크

### 21.1 서버 truth와 UI truth의 분리

가장 큰 문제는 이것이다.

- 서버에는 `watch_jobs`가 쌓이는데
- 메인 UI는 서버 목록을 읽지 않고 `localStorage`를 진실 원천처럼 사용한다

결과:

- DB와 브라우저 목록이 어긋날 수 있음
- 다른 브라우저/기기에서 같은 계정 개념 없이 일관성이 깨짐
- 워커는 DB를 기준으로 동작하는데 화면은 로컬 목록을 기준으로 보일 수 있음

### 21.2 활성/대체/레거시 구현 공존

같은 `/watch`, `/click` 개념이 세 방식으로 존재한다.

- Next `app/api`
- `render-api`
- Cloudflare worker

여기에 `legacy-api`까지 있어 유지보수 혼란이 크다.

### 21.3 Coupang 지원 불일치

지원 상태가 파일마다 다르다.

- README: 지원하는 듯 보임
- URL parser: 입력 차단
- batch: 일부 지원
- worker: 중단

이 상태는 제품 메시지와 실제 동작을 어긋나게 만든다.

### 21.4 느린 watch 생성 API

`POST /api/watch`가 초기 Amazon 수집까지 기다리므로:

- 응답 지연 가능성 큼
- 외부 사이트 차단/지연에 직접 영향
- 사용자 등록 UX 불안정 가능성

### 21.5 환경 이름 혼선

- `ALLOWED_ORIGINS`
- `ALLOWED_ORIGIN`
- `NEXT_PUBLIC_API_BASE_URL`
- `NEXT_PUBLIC_WORKER_API_BASE_URL`
- `NEXT_PUBLIC_CLICK_API_BASE_URL`

특히 `watch-dashboard.js`가 watch 엔드포인트 후보로 `NEXT_PUBLIC_CLICK_API_BASE_URL`까지 참조하는 점은 의미상 혼란스럽다.

### 21.6 개발 환경 드리프트

- `.idx/dev.nix`: Node 20
- `render.yaml`: Node 22

심각한 문제는 아닐 수 있지만 재현성 차이는 발생할 수 있다.

## 22. 이 프로젝트를 한 문장으로 요약하면

현재 이 저장소는:

“Amazon URL 감시를 중심으로 한 가격 추적 MVP가 Next.js UI, PostgreSQL, HTML 스크레이핑 워커, 웹훅/이메일 알림 구조 위에 부분적으로 완성되어 있으나, 서버 상태와 브라우저 상태가 분리되어 있고, 대체/레거시 구현이 함께 남아 있어 실제 운영 구조가 문서보다 더 분산된 프로젝트”

라고 정리할 수 있다.

## 23. 추천 우선순위

현재 코드를 바꾸지 않고 구조 이해만 기준으로 봤을 때, 가장 먼저 손봐야 할 우선순위는 아래와 같다.

1. 메인 UI를 DB 기반 watch 목록 조회와 연결
2. `Next API` / `render-api` / `Cloudflare worker` 중 하나를 기준 구현으로 확정
3. Coupang 지원 상태를 문서/코드/UX에서 일치시킴
4. `POST /api/watch`에서 초기 수집을 비동기화하거나 워커로 분리
5. 문서에서 “현재 실제 구조”와 “권장 구조/과거 구조”를 구분

## 24. 최종 판단

기초 골격은 나쁘지 않다. 특히 아래는 이미 갖춰져 있다.

- DB 스키마
- watch 개념
- 수집 배치와 감시 워커
- 환율 캐시
- 알림 이력 기록
- 웹훅/이메일 알림

하지만 현재는 “기능이 없는 상태”라기보다 “여러 세대의 구현이 겹쳐 있고, 메인 UI와 서버 상태 연결이 덜 마무리된 상태”에 더 가깝다.

따라서 이 프로젝트를 이해하는 핵심은 단순히 파일 수를 많이 읽는 것이 아니라, 어떤 구현이 현재 실제 동작 경로인지 구분하는 것이다. 이번 조사 기준으로 그 실제 중심선은 다음과 같다.

- 입력 UI: `components/watch-dashboard.js`
- 저장 API: `app/api/watch/route.js`
- 핵심 서비스: `lib/watch-service.js`
- 감시 실행기: `collector/run-watch-worker.js`
- 영속화 중심 테이블: `watch_jobs`, `notifications`, `products`, `price_history`
